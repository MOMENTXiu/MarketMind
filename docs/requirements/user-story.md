# 超市 AI 营销系统 User Story

## 1. 背景

作为超市 IT 经理，我需要带领团队建设一套后端驱动的 AI 营销分析系统，把门店销售明细转化为可执行的营销决策。系统应基于新版 `analysis/` 的处理逻辑实现，不再沿用旧版简化分析链路，也不以命令行脚本作为运行方式。

本阶段明确排除 TTS / 语音播报能力。系统只需要产出结构化分析结果、图表/表格/报告类 artifact、推荐结果和营销策略。

## 2. 原始需求重写

原始需求：

1. 制定某些商品的促销策略：需要合并同个订单 ID 的记录，形成购物清单集，找出商品作为后项的关联规则。
2. 预测未来的销售额和利润。
3. 对客户聚类分析，制定针对性的营销策略。

新版系统需求：

1. 系统应读取超市原始销售明细，完成数据清洗、字段标准化、退货标记、促销字段映射和数据质量报告。
2. 系统应构建顾客画像、商品画像、复购周期、价格带、促销敏感度等特征，作为后续分析基础。
3. 系统应基于购物篮和商品层级关系挖掘关联规则和高效用组合，生成可执行的商品组合促销策略。
4. 系统应预测销售趋势，并支持面向经营决策的时序、价格弹性、品类增长和促销效果分析。
5. 系统应完成顾客分群，生成群体画像、群体价值排序和差异化营销策略。
6. 系统应生成消费者侧个性化推荐和营销者侧经营洞察，包括推荐理由、组合策略、发券建议、流失风险、CLV 和品类经营建议。

## 3. 角色

| Role | Description |
|---|---|
| IT 经理 | 负责系统建设、数据接入、后端流程、权限和交付质量。 |
| 营销经理 | 使用系统制定促销组合、发券策略、客群触达和品类经营动作。 |
| 门店管理层 | 查看经营摘要、销售趋势、客户结构和营销策略优先级。 |
| 数据分析师 | 校验模型结果、调整参数、解释算法结论和评估效果。 |
| 后端服务 | 执行数据处理、分析编排、模型训练、artifact 存储和 API 输出。 |

## 4. Epic

### Epic 1: Retail 数据准备

作为 IT 经理，我希望系统能够上传并识别超市销售明细，自动完成清洗和质量诊断，以便后续分析建立在可信数据上。

Acceptance criteria:

- 系统支持原始销售 CSV，字段包含顾客编号、商品编码、类目编码/名称、销售日期、销售金额、销售数量、商品单价、是否促销等。
- 系统能处理 GBK / UTF-8-SIG 编码。
- 系统能识别并修复可恢复的错位行、重复行、非法日期、缺失规格、异常单价和单位脏值。
- 系统能标记退货/异常交易，并在购物篮、推荐和正向销售分析中排除。
- 系统输出 clean dataset、数据质量摘要和字段映射结果。

### Epic 2: 顾客与商品画像

作为营销经理，我希望系统自动生成顾客画像、商品画像和复购周期，以便理解谁在买、买什么、多久复购、对价格和促销是否敏感。

Acceptance criteria:

- 系统生成顾客画像，包含 RFM、类目偏好、类目熵、促销敏感度、生鲜占比、价格偏好、退货率等指标。
- 系统生成商品画像，包含销售金额、销售数量、购买人数、复购率、促销率、价格带和综合热度。
- 系统生成复购周期与复购紧迫度，支持推荐和召回策略使用。
- 系统使用 CRITIC / TOPSIS 等方法生成可解释的综合得分。

### Epic 3: 商品组合促销策略

作为营销经理，我希望系统根据购物篮挖掘商品关联规则和高效用组合，以便制定组合促销、第二件折扣、跨品类联动和陈列策略。

Acceptance criteria:

- 系统按顾客和销售日期聚合购物篮。
- 系统支持 item、小类、中类等层级的 FP-Growth 关联规则。
- 系统输出支持度、置信度、提升度、前项、后项和策略建议。
- 系统支持“后项商品”查询，帮助寻找适合被带动销售的商品。
- 系统支持高效用项集，发现低频但高客单价的组合。
- 系统输出 bundle strategy，包含组合价值得分、促销提升度和动作建议。

### Epic 4: 顾客分群与营销策略

作为营销经理，我希望系统对顾客进行分群并生成每个群体的经营策略，以便针对不同客户群体做差异化运营。

Acceptance criteria:

- 系统支持 GMM、UMAP + HDBSCAN、AE + GMM 等分群思想，最终产出可落地群体。
- 系统输出群体人数、人数占比、R/F/M、促销敏感度、生鲜占比、类目熵、低价带偏好和销售贡献。
- 系统为群体生成业务语义标签，例如高价值稳定型、促销敏感型、流失预警型、生鲜高频型、类目集中型。
- 系统输出群体价值排序，帮助确定营销资源优先级。

### Epic 5: 消费者侧个性化推荐

作为营销经理，我希望系统为顾客生成可解释的商品推荐，以便支持会员推荐、精准发券和商品触达。

Acceptance criteria:

- 系统支持热门、类目偏好、关联规则、图嵌入、复购周期、促销适配、价格偏好等多召回信号。
- 系统使用 CRITIC-TOPSIS 或等价融合策略进行排序。
- 系统输出 Top-K 推荐商品、分数、来源信号和推荐理由。
- 系统评估 Precision、Recall、HitRate、NDCG、Coverage、Diversity 和 PromoMatchRate 等指标。
- LightGCN 可作为高级推荐能力进入，但不得阻塞基础推荐链路。

### Epic 6: 销售预测与经营趋势

作为门店管理层，我希望系统预测未来销售趋势并解释季节性和波动，以便安排库存、补货、排班和促销档期。

Acceptance criteria:

- 系统按日期聚合销售金额，支持季节分解、趋势分析和回测。
- 系统输出未来销售预测、回测指标和可视化 artifact。
- 系统支持销售额和利润相关的经营指标输出。
- 系统能在样本不足或日期不连续时返回明确错误。

### Epic 7: 营销者侧因果与策略洞察

作为营销经理，我希望系统区分“看似有效”和“真正有效”的促销策略，以便减少全员撒券和无效降价。

Acceptance criteria:

- 系统支持 DML 估计促销 ATE / CATE，控制价格、品类、时间和顾客特征等混淆因素。
- 系统输出朴素均值差和去偏因果效应对比。
- 系统识别对促销真正正响应的人群。
- 系统支持 uplift 发券名单、流失预测、CLV、价格弹性和品类经营象限作为高级分析。
- 系统输出营销者侧结构化洞察和报告 artifact。

### Epic 8: 后端化运行与交付

作为 IT 经理，我希望所有分析能力以可测试的后端服务运行，而不是依赖命令行脚本，以便系统可以稳定集成到前端、任务调度和后续运营系统。

Acceptance criteria:

- API Controller 只负责请求校验、调用 Flow/Pipeline、响应映射和错误映射。
- Business Flow 管理完整 Retail 分析生命周期。
- Business Pipeline 编排阶段逻辑。
- Ability Atom 实现最小业务动作。
- Provider Interface 隔离数据集、artifact、模型存储和任务调度。
- External Adapter 负责 CSV、文件、模型序列化、图表保存等基础设施细节。
- 后端运行路径不得直接 import `analysis/code_files/*.py`。
- 系统不提供 TTS / 语音播报能力。

## 5. Out Of Scope

- TTS / 语音播报 / 音频生成。
- 旧版 Superstore 字段和旧分析 API 的兼容要求。
- 直接运行 `analysis/code_files/*.py` 的命令行工作流。
- 将 `analysis/output` 作为后端运行时 source of truth。
- 在未做 dependency gating 前强制启用 LightGCN、SHAP、Lifetimes 等重依赖能力。

## 6. Definition Of Done

- SRS 覆盖功能需求、非功能需求、数据契约、API 契约、错误语义和验收标准。
- 新版 Retail V2 分析链路有测试锚点。
- 后端四层边界通过 Architecture Lint。
- Runtime Check 能执行 sample dry-run。
- 文档明确 TTS 不在范围内。
